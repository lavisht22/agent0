## Guiding principles

1. **Data stays put, reversible steps.** Supabase's Postgres remains the source of truth. We swap *access layers*, not rows, until the final phase. Any step can be rolled back without data loss.
2. **The Supabase SDK and a direct Postgres/Drizzle connection coexist** against the same database. Migration off the SDK happens table-by-table, not in a big bang.
3. **One principal, many credentials.** Authenticate at the edge; normalize every credential (browser session, PAT, API key) into a single `Principal`. Route handlers depend only on `principal.scopes`, never on *how* the caller authenticated. (See [Auth architecture](#auth-architecture).)
4. ~~**RLS goes dormant, not deleted.**~~ **REVISED 2026-06-06 ‚Äî RLS removed.** The original plan kept RLS as a dormant backstop. Once Phases 1‚Äì2 landed, RLS only ever constrained the `anon`/`authenticated` roles (Supabase Data API + old direct-browser access), and both runner paths (`postgres`/`service_role`) have `rolbypassrls = true` ‚Äî so it was fully dormant. With the user **switching off the Supabase Data API** (removing the only entry point for those roles) and wanting a clean vanilla-Postgres end state, we **dropped all 39 policies + disabled RLS on all 15 tables** (migration `20260606120000_drop_rls_policies.sql`). The `is_workspace_*` helper functions are KEPT (the live `workspace_assign_owner_admin` trigger still calls `is_workspace_admin`); the `auth.uid()` column defaults stay until the Phase-4 DB move.

## Background: what Supabase does for us

Four distinct platform features, each with a different switching cost:

| Feature | Where it's used | Switching cost |
|---|---|---|
| **Postgres** (the data) | everywhere | Low ‚Äî it's just Postgres |
| **Auth** (`signInWithOtp`, `verifyOtp`, `getSession`, `getClaims`, `getUser`, `admin.getUserById`) | web + runner | Medium ‚Äî OTP / magic-link email auth |
| **Storage** (run-log blobs in the `runs-data` bucket, keyed by `runId`) | `helpers.ts`, `runs.ts`, `queries.ts` | Low‚ÄìMedium |
| **RLS** (`is_workspace_reader/writer/admin`, `auth.uid()`) | ~all tables | High ‚Äî but it goes *dormant*, see principle #4 |

The center of gravity is **not** the database swap (Postgres is portable). It's that the **web app talks directly to Postgres from the browser** with the anon key (~40 `.from("table")` calls), with *all* authorization living in RLS tied to `auth.uid()`. Moving that behind the runner API ‚Äî and re-implementing RLS rules in code ‚Äî is the bulk of the work and the bulk of the risk.

## Auth architecture

### The standard we're following

Multiple credential types against one API is normal (GitHub/Stripe/GitLab). The discipline is: **authenticate ‚Üí normalize to one `Principal` ‚Üí authorize on the principal only.**

```ts
type Principal =
  | { kind: "user";   userId: string; workspaceId?: string; scopes: string[] }              // browser session OR PAT
  | { kind: "apiKey"; workspaceId: string; scopes: string[]; allowedOrigins: string[] | null }
```

The auth middleware is an **ordered list of authenticators** (Passport-style strategies); first one to yield a `Principal` wins. **Discrimination is purely header-keyed** (revised 2026-06-05 ‚Äî no prefix-sniffing on `Authorization`): each credential type has its own header lane.

1. **Browser session** ‚Üí `kind: "user"`. `Authorization: Bearer <session-token>` (Supabase JWT in Phase 1; better-auth **bearer-plugin session token** from Phase 2). Selected whenever the `Authorization` header is present ‚Äî this header is now *exclusively* the browser session's.
2. **PAT** (`agent0_pat_‚Ä¶`) ‚Üí `kind: "user"`. **`x-pat: agent0_pat_‚Ä¶`** (dedicated header, revised from the old `Authorization: Bearer` + prefix). CLI; inherits the user's *current* workspace role dynamically (demote the user ‚Üí their PATs weaken on next request). **Stays an agent0-owned table** (`personal_access_tokens`) ‚Äî better-auth is not involved. The `agent0_pat_` prefix is retained on the token value for secret-scanning only, not routing.
3. **API key** ‚Üí `kind: "apiKey"`. `x-api-key: ‚Ä¶`. Machine identity; workspace-pinned, fixed scopes, origin allowlist. Stays an agent0-owned table (`api_keys`).

Route handlers read only `principal.scopes`. `kind === "user"` replaces today's `requireUserId` for mutations that must exclude machine keys. PAT and API key stay **distinct kinds** (different semantics) but produce one normalized type.

### Browser credential: bearer token in `Authorization` (decided), not cookies

The browser keeps sending `Authorization: Bearer <token>` ‚Äî it already does this today with the Supabase access token. From Phase 2 the token is better-auth's **bearer-plugin session token** (opaque, DB-backed), **not** a signed JWT. Rationale:

- **Minimal churn** ‚Äî Phase 1‚Üí2 only swaps the token *issuer/validator*; the transport is unchanged. No cookie/CSRF/CORS-with-credentials machinery to add or remove.
- **Uniform model** ‚Äî everything is a header/bearer token across browser, CLI, and machines.
- **No CSRF surface** ‚Äî `Authorization` isn't auto-sent by the browser (cookies are); and no cross-origin cookie pain in dev (:2222 ‚Üî :2223) or for the embed use case.
- **Discrimination stays clean** ‚Äî PAT is already `agent0_pat_`-prefixed, so "session vs PAT" on the same header is an O(1) prefix check.

**Bearer-plugin session token vs JWT plugin:** better-auth's *bearer plugin* sends an opaque, DB-backed session token (instant revocation, no signing-key infra) ‚Äî the right default for our stateful runner that already hits Postgres every request. better-auth's *JWT plugin* (signed JWTs verified via JWKS, no DB hit) is explicitly "not a replacement for the session" and is for stateless/external verification; keep it in reserve, only adopt if a future need for stateless verification appears.

**Tradeoff accepted:** a bearer token held in JS is exfiltratable by XSS (an httpOnly cookie wouldn't be). Mitigate with short-lived sessions + refresh, keeping the token **in memory (not localStorage)**, and a strict CSP. (See Risks.)

### Today's wiring (for reference)

- `/internal/test`, `/internal/refresh-mcp` ‚Äî validate the Supabase JWT via `supabase.auth.getClaims(token)`, registered **outside** `addAuth`.
- `/api/v1/workspaces/:workspaceId/*` ‚Äî go through `addAuth`, which only knows **PAT** (`Bearer`, hashed ‚Üí `personal_access_tokens`) and **API key** (`x-api-key`). The web app does **not** use these yet.

- [x] **D6 ‚Äî Scheduled jobs** (`delete_old_runs`). ‚úÖ **Removed, not ported (2026-06-07, with user).** The auto-delete-old-runs feature is dropped entirely rather than re-homed. Findings that drove the call: the live DB had **no `cron.job` scheduled** ‚Äî the `delete_old_runs()` SECURITY-DEFINER function existed but nothing invoked it, so the feature was already dormant (deleting nothing). Re-implementing it as a runner job would have needed single-execution coordination under a load-balanced multi-runner deploy (advisory lock / lease / dedicated scheduler), and `pg_cron` was rejected because it couples self-host to a Postgres build with the extension preloaded ‚Äî against the "just a Postgres host" goal. **Done:** dropped the live function (`DROP FUNCTION public.delete_old_runs()`), deleted its migration (`20251215064234_delete_old_runs.sql`), removed the generated `delete_old_runs` RPC type from `database.types.ts`. **Future:** retention becomes a **user-controlled manual feature in the dashboard** (the user picks what/how much older data to delete) rather than an implicit background purge.
- [x] **D13 ‚Äî Secret encryption scheme.** ‚úÖ **Client-side PGP (asymmetric) ‚Üí runner-side AES-256-GCM (symmetric AEAD), one-shot re-encrypt (2026-06-07, with user).** Provider/MCP config secrets must be stored **reversibly encrypted** (the runner needs plaintext to call providers ‚Äî can't hash) so a DB dump / dashboard access / RLS slip is useless without the app key. The original design used **PGP** purely because the **untrusted browser was the writer** (direct `.from("providers").insert` under the anon key): asymmetric crypto let the browser hold only the **public key** (encrypt-only, shipped to every browser) while only the runner held the private key (decrypt at run time). **That precondition is gone** ‚Äî since Phase 1 every write goes through the runner API, and the runner is a fully-trusted boundary (already holds the DB creds, S3 keys, *and* the PGP private key, and round-trips other secrets ‚Äî minted API keys, raw PATs ‚Äî in plaintext over the same TLS channel). So the asymmetric split now buys **zero** security, only complexity. **Switch to symmetric:** the browser sends **plaintext config over TLS**; the runner encrypts on create/update and decrypts on the run path with **AES-256-GCM** via Node's built-in `crypto` (no dependency). One 32-byte key in env (`CONFIG_ENCRYPTION_KEY`), runner-only. Threat model unchanged (rows still hold only ciphertext). **Drops:** `openpgp` + `@openpgp/web-stream-tools` from **both** apps, the hand-written `apps/runner/src/lib/openpgp.{js,d.ts}` + its `cp` build step, the `VITE_PUBLIC_PGP_PUBLIC_KEY` / `PGP_PRIVATE_KEY` / `PGP_PRIVATE_KEY_PASSPHRASE` env, and the client-side `encryptConfig()` in both web routes ‚Äî and removes openpgp's wasm/eval as a CSP `'unsafe-eval'`/CDN driver (Monaco remains the only one). **Migration = one-shot re-encrypt** (chosen over lazy dual-read, with user): a runner script PGP-decrypts every `providers`/`mcps` row ‚Üí AES-encrypts ‚Üí writes back, format-detected so it's idempotent. See **Phase 6**. Rejected alternatives: disk/TDE & `pgcrypto` (don't defend the actual threat ‚Äî DB-read access ‚Äî and/or put the key in the DB); KMS/Vault (extra infra, fights the bundled-self-host goal). DB columns keep the name `encrypted_data_*` (still accurate ‚Äî runner-encrypted at rest).
- [x] **D12 ‚Äî Casing standard.** ‚úÖ **snake_case end-to-end** (2026-06-06). Drizzle app-table properties named snake_case (no column-name strings); one casing across DB/Drizzle/JSON, no translation layer, no alias maps. better-auth tables stay camelCase. Stripe/GitHub-style; matches the existing wire contract. Value-format normalization (ISO/number) stays per-handler for now.

**Status:** ‚úÖ **CODE COMPLETE (2026-06-07) ‚Äî prod cutover pending.** Simplifies away the asymmetric-PGP machinery that only existed because the *browser* used to be the (untrusted) writer ‚Äî see **D13** for the full rationale. ## Risks & notes

- **Authz parity is the #1 risk.** RLS is declarative + Postgres-enforced; moving it into runner code per-endpoint must replicate `is_workspace_reader/writer/admin` exactly. Dedicated security review after Phase 1b/1c and after Phase 2's middleware swap.
- **Browser bearer-token XSS exposure** (D8 tradeoff) ‚Äî a bearer token in JS can be exfiltrated by XSS. Required mitigations: short-lived sessions + refresh rotation, token held **in memory (not localStorage)**, strict CSP. This is the price of not using httpOnly cookies; treat the mitigations as non-optional.
- **UUID preservation** (D10) is the correctness lynchpin of the auth migration.
- **Re-login at auth cutover** ‚Äî all sessions invalidate in Phase 2; communicate to users.
- **Pooler/prepared-statement caveat** when connecting through Supabase's transaction pooler (Phase 2/3).
- **Email delivery** (D7) becomes our responsibility in Phase 2.
- **CLI** (`packages/cli`) talks to the runner API, not Supabase ‚Äî largely unaffected, but re-verify after Phase 2 (token prefixes/auth) and Phase 3.
- **`@repo/database` contract**: keep its export shape stable across the SDK‚ÜíDrizzle swap to limit churn in web/runner/cli.
- **Phase 6 cutover is order-sensitive (all-at-once via maintenance window).** The chosen approach has no transitional dual-format code, so the AES-only runtime must **not** be serving while any PGP rows remain (and the old PGP-only runtime must not serve after rows flip to AES). Required order at cutover: **stop old runner ‚Üí run `reencrypt-configs` ‚Üí start new AES-only runner + plaintext web.** Running the script while the old PGP-only runtime is live would break reads as rows flip; deploying the new runtime before the script would break reads of still-PGP rows. **`CONFIG_ENCRYPTION_KEY` is now a load-bearing secret** ‚Äî it must be set in the prod runner env *before* cutover, and losing it makes every provider/MCP config unrecoverable (same property PGP had; back it up). Keep `PGP_PRIVATE_KEY`/passphrase set until the re-encrypt is confirmed done (the script needs them; they're the only way to read pre-migration blobs), then remove in the follow-up cleanup commit.
